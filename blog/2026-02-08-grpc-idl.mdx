# gRPC 进阶 (0)：Protobuf 语法与 IDL 声明全解析

> “API 设计是微服务的地基，而 Protobuf 就是那支最锋利的画笔。”

在深入探讨 [HTTP/2 协议](/blog/grpc-protocol) 和 [Java 生产实践](/blog/grpc-production) 之前，我们必须先掌握 gRPC 的核心语言——**Protobuf (Protocol Buffers)**。它不仅决定了数据如何序列化，更定义了服务的边界。

本文将作为系列零章，带你从零到一掌握 Protobuf 的每一处常用声明及其用法。

## 一、 文件的艺术：基础声明

每个 `.proto` 文件都应以基础配置开始。

### 1.1 语法版本

```protobuf
syntax = "proto3";
```

目前的标准是 `proto3`。它移除了 `required` 字段（避免了臭名昭著的破坏性更改），并将默认值设为零值，极大地简化了代码生成及处理逻辑。

### 1.2 包名与导入

```protobuf
package com.example.v1;

import "google/protobuf/timestamp.proto";
import "other_service.proto";
```

- **package**：防止命名冲突，转换到 Java 时默认会作为 `java_package` 的基础。
- **import**：引入公共定义或其他服务的定义。

### 1.3 常用选项 (Options)

```protobuf
option java_multiple_files = true;
option java_package = "com.example.grpc.generated";
option go_package = "example.com/api/v1";
```

这些选项影响代码生成器（protoc）的行为。例如 `java_multiple_files` 能够让编译器为每个 Message 生成独立的 Java 类，而不是全部塞进一个巨大的内部类中。

## 二、 数据结构：Message 的深度用法

`message` 是 Protobuf 的核心，类似 Java 的 Class 或 Go 的 Struct。

### 2.1 字段编号 (Field Tags)

```protobuf
message User {
  int32 id = 1;
  string name = 2;
  string email = 3;
}
```

**关键点**：`1` 到 `15` 的编号在二进制编码中仅占 1 个字节，应留给**高频访问**的字段。**编号一旦分配，严禁修改。**

### 2.2 集合与映射

```protobuf
message Order {
  repeated string item_ids = 1; // 列表/数组
  map<string, string> metadata = 2; // Key-Value 映射
}
```

`repeated` 是列表的声明方式。`map` 的 Key 只能是基本类型（不能是 Message）。

### 2.3 枚举 (Enum)

```protobuf
enum OrderStatus {
  UNKNOWN = 0; // 第一个常量必须映射到 0
  PENDING = 1;
  SHIPPED = 2;
  CANCELLED = 3;
}
```

**避坑指南**：Protobuf 要求第一个枚举值必须为 0，作为缺省值。

### 2.4 包裹类型与 Optional

在 `proto3` 中，基本类型如果不赋值，会默认序列化为默认值（如 0 或 ""）。如果需要区分“未设置”和“零值”，推荐使用 `google.protobuf.Int32Value` 等 Wrapper 类型。

## 三、 服务契约：Service 与 RPC

这定义了服务的“动作”。

```protobuf
service OrderService {
  // 1. 一问一答 (Unary RPC)
  rpc GetOrder(OrderRequest) returns (OrderResponse);

  // 2. 服务端流 (Server Streaming)
  rpc WatchOrders(OrderRequest) returns (stream OrderUpdate);

  // 3. 客户端流 (Client Streaming)
  rpc BulkUpload(stream OrderRequest) returns (UploadSummary);

  // 4. 双向流 (Bidirectional Streaming)
  rpc Chat(stream Message) returns (stream Message);
}
```

- **Unary**：最常用的模式，类似 REST。
- **Streaming**：适用于推送通知、大文件上传或实时聊天。

## 四、 最佳实践：版本演进

API 永远在变化，如何保持向前兼容？

1.  **绝不修改字段编号**。
2.  **不删除已存在的字段**：改用 `reserved` 关键字保留编号，防止后来的开发者复用，导致数据污染。
    ```protobuf
    message User {
      reserved 2, 4 to 6;
      reserved "old_field", "obsolete_field";
    }
    ```
3.  **使用 `v1`, `v2` 包路径**：在重大破坏性更新时，通过包路径进行版本隔离。
